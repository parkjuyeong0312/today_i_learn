# export db를 따로 둬야할까?
방식은 2가지를 생각해볼 수 있을 것 같다.

### 1. 클라이언트에서 변환/ 캐싱 X/ DB 저장안함
- 클라이언트에서 html2canvas 같은 라이브러리를 사용해서 자바스크립트 상의 정보를 그대로 긁어와서 사용하는 방식.
- 서버에 요청을 보내는 방식이 아니기 때문에, 그대로 클라이언트에게 반환한다.
- 유저가 다운로드 하면 되고, 서버에 api 요청을 하지 않기 때문에, 캐싱을 하지 않아도 된다. 당연히 DB 저장도 하지 않는다.

### 2. 서버측에서 변환 / 캐싱 O
- 그럼 이미지 파일을 저장하는 걸까?
- 그렇다면 S3와 같은, 파일 저장소 NoSQL을 사용해야됨.
- 만약 하게된다면, Redis에 room_id+updated_at의 조합으로 키를 생성할 것 같다.
- 흐름
    - 유저 export 요청(room_id) -> 서버가 해당 room_id의 updated_at확인 -> Redis에 캐시 있나 key값으로 확인 -> 있으면 S3 url반환, 없으면 이미지 생성해서 S3에 저장해놓고, Redis에 키+S3 URL에 저장

> 구글 url같은 경우, 구글서버에 이미지 요청을 하면 구글서버의 S3에서 이미지를 반환해주는 형식인데, 결과물을 저장해야되는 입장에서는 S3에 저장해야한다.

### 나의 선택!
캐싱 이점 < 구현 복잡도 + S3 관리 비용  
이므로 그냥 클라이언트딴에서 출력하는게 처리가능할 것 같다.  
그냥 이미지 출력방식이아니라, Ai 분석 결과물이라면, api 요청에 대한 비용 부담이 발생하니, 이 부분에 대해서는, S3를 고려해볼 필요도 있을 것 같다.
