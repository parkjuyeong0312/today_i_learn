# 장소카드 메세지 전송은 http방식으로 해야할까? 웹소켓으로 해야할까?

#### 문제상황
여행 일정을 함께 짜는 유저들이 장소카드를 채팅에 공유할 때,
카카오톡 장소공유처럼 실시간으로 모든 유저에게 전달되어야 한다.
하지만 이 장소카드를 채팅방에 공유하는 방식을 HTTP로 해야할까? 웹소켓으로 해야할까에 대한 고민을 담은 글이다.

### 선택지

#### 방법 1. HTTP + WebSocket 브로드캐스팅
- 유저 A가 채팅창에 공유하고 싶은 정보를 서버에 post매서드로 전송
- 서버가 DB에 장소카드, 저장
- 서버가 redis에 publish
- subscriber 서버에게 전달
- 연결된 유저들에게 브로드캐스팅


#### 방법 2. WebSocket형식으로 전송

**흐름:**
1. 유저 A가 WebSocket으로 장소카드 전송
2. 서버가 DB 저장 (일반 채팅 메시지와 같은 테이블, `type` 컬럼으로 구분)
3. Redis publish → Subscriber 서버 수신
4. 채팅방 유저 전원에게 브로드캐스팅

> 방법 2가 적절해보임.

**DB 설계:**

| 컬럼 | 설명 |
|------|------|
| `id` | PK |
| `room_id` | 채팅방 |
| `sender_id` | 보낸 유저 |
| `type` | `TEXT` / `PLACE_CARD` |
| `content` | 텍스트면 메시지 내용, 장소카드면 장소 데이터 JSON 또는 FK |
| `created_at` | 전송시각 |

> content면, 이 내부 구조를 구체화할 필요가 있어보임.

**지도 포커싱 연동:**
- 우리 서비스는 같은 페이지에 지도와 채팅창이 함께 존재
- 채팅창에서 장소카드 클릭 시, 서버 추가 요청 없이 클라이언트 내부 상태(전역 상태)로 지도 컴포넌트에 좌표 전달
- 지도 컴포넌트가 해당 좌표로 포커싱 + 마커 강조
  채팅 컴포넌트: dispatch(focusMap({ lat, lng }))
  지도 컴포넌트: focusMap 상태 변경 감지 → 해당 좌표로 이동
- 따라서 payload에 `latitude`, `longitude` 값이 반드시 포함되어야 함

**카카오톡과의 차이:**
- 카카오톡 장소공유: URL + OGTag(썸네일, 제목, 설명) 렌더링 방식
- 우리 서비스: URL 없이 카카오맵 API 등에서 받아온 장소 데이터를 payload로 직접 전송 → 클라이언트가 `type === "PLACE_CARD"`일 때 전용 카드 UI 렌더링
> 우리는 OG태그를 쓸 필요는 없음.
