## NoSQL이란?
NoSQL 데이터베이스는 'Not Only SQL'이라는 의미  
관계형 테이블과는 다른 방식으로 데이터를 저장함.

## NoSQL 데이터베이스란?
보통 'NoSQL 데이터 베이스'라는 용어는 비 관계형 데이터베이스를 지칭할 때 사용됨.

### 데이터베이스 유형(4가지)
- 문서형 데이터베이스
- 키-값 형 데이터베이스
- 와이드 컬럼형 저장소
- 그래프 데이터베이스

#### 1. 문서지향 데이터베이스
- 문서 지향 데이터베이스는 **JSON 객체**와 유사한 문서 형태로 데이터를 저장함.  
- 각 문서는 필드-값 쌍 을 포함함.  
- 반 정형 데이터나 비 정형 데이터 집합에 적합함.  
- 중첩된 구조를 지원하여 계층적 데이터 표현 가능.

대표적인 예시 : MongoDB, Couchbase

```json
{
  "_id": "12345",
  "name": "foo bar",
  "email": "foo@bar.com",
  "address": {
    "street": "123 foo street",
    "city": "some city",
    "state": "some state",
    "zip": "123456"
  },
  "hobbies": ["music", "guitar", "reading"]
}
```


#### 2. 키-값 데이터베이스
- 키와 값으로 이루어진 단순한 데이터베이스
- 캐싱과 세션관리에 주로 사용됨
- 메모리에 데이터를 저장하기에 읽기 및 쓰기 성능이 좋음.

대표적인 예시 : Amazon DynamoDB, Redis

```json
    Key: user:12345
    Value: {"name": "foo bar", "email": "foo@bar.com", "designation": "software developer"}
```

### NoSQL 데이터베이스 유형 비교
| 비교 항목 | 키-값 (Key-Value) DB (예: Redis) | 문서 (Document) DB | 와이드 컬럼 (Wide-Column) DB | 그래프 (Graph) DB |
|---|---|---|---|---|
| 핵심 구조 | 단순 키-값(Key-Value) 쌍 | JSON/BSON 문서 | 컬럼 및 로우 패밀리(Row families) | 노드(Node), 간선(Edge), 관계 |
| 스키마 | 스키마리스 (Schemaless) | 유연한 스키마 | 유연한 스키마 | 유연한 스키마 |
| 쿼리 제약 | 키(Key)에 의해서만 제한됨 | 풍부한 쿼리 기능 지원 | 컬럼에 의해 제한됨 | 특화된 그래프 쿼리 지원 |
| 인덱싱 | 단순 인덱스 | 필드(Fields) 기반 인덱스 | 보조 인덱스 (Secondary indexes) | 노드/간선 기반 인덱스 |
| 확장성 및 일관성 | 수평적 확장성 및 결과적 일관성 (공통) | 수평적 확장성 및 결과적 일관성 (공통) | 수평적 확장성 및 결과적 일관성 (공통) | 수평적 확장성 및 결과적 일관성 (공통) |
| 분석 적합성 | 복잡한 분석 지원 제한적 | 시계열, IoT 분석, 실시간 분석에 적합 | 시계열 데이터, IoT 분석에 적합 | 그래프 분석에 최적화됨 |
| 데이터 유형 | 모든 유형의 데이터 저장 가능 | JSON/BSON 데이터 저장 | 컬럼과 로우 매트릭스(Matrix) | 노드, 간선, 관계 |
| 데이터 조작 (CRUD) | 기본 CRUD 작업, 제한적인 데이터 조작 기능 | 고급 쿼리 및 데이터 조작 기능 | 데이터 조작 및 CRUD 기능 | 고급 그래프 탐색 및 조작 기능 지원 |
| 검색 기능 | 기본적인 키 기반 데이터 검색 | 풀텍스트(Full-text) 검색, 벡터(Vector) 검색 | 제한적인 검색 기능 | 제한적인 검색 기능 |
| 데이터 보관 (Archival) | 온라인 데이터 보관 지원 제한적 | 자동 온라인 데이터 보관 지원 | 온라인 데이터 보관 지원 제한적 | 온라인 데이터 보관 지원 제한적 |
| 보안 및 접근제어 | 기본 인증, 제한적인 접근 제어 | 역할 기반 접근 제어(RBAC), 미사용/전송중/사용중 암호화 | 역할 기반 접근 제어(RBAC), 미사용(at rest) 암호화 | 역할 기반 접근 제어(RBAC), 미사용 및 전송중 암호화 |
| 데이터 시각화 | 기본 지원 | 기본 지원 | 기본 지원 | 기본 지원 |



### NoSQL 데이터베이스의 역사
- 스토리지 비용이 급격하게 감소하기 시작할 때, 등장함.
- 데이터 중복을 피하고자 어려운 모델을 만들 필요가 없어짐. 
- 데이터의 종류도 많아짐. 스키마를 사전에 정의하는 것은 불가능함.

### NoSQL 데이터베이스의 특징
- 분산 컴퓨팅
- 확장성
- 유연한 스키마와 풍부한 쿼리 언어
- 개발자 친화적인 사용 편의성
- 파티션 내결함성
- 높은 가용성

#### BASE 원칙
- Basically Available (기본 가용성): "일부 서버가 죽어도, 서비스는 멈추지 않는다"  
    - 완벽하게 동기화가 안되어도 응답을 일단 주는데 초점.
- Soft state (소프트 상태): "상태는 고정되어 있지 않고 변할 수 있다"
    - 데이터 상태가 고정되어 있지 않고 유연하게 변할 수 있음.
- Eventual Consistency (궁극적 일관성): "당장은 다를 수 있어도, 결국에는 같아진다"  
    - 당장의 데이터 일관성은 없더라도, 결국 나중에는 동기화처리가 되어 같아짐.
