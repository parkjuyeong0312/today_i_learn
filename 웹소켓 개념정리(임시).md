원래는, 클라이언트가 서버에게 말을 걸어주는 방식임.
서버는 클라이언트가 말을 걸어주지 않으면 응답을 못함.
이게 보통 http 통신 방식

이것을 반 이중 통신이라고함(half duplex communication)

근데 클라이언트의 요청 없이도 서버에서 알아서 데이터를 보내줄 수는 없는걸까?

## 통신방식
### 1.Polling
클라이언트가 서버로 주기적으로 요청을 보내는 방식

### 2. Long Polling
클라이언트가 요청을 걸어두는 방식
만약 서버에 데이터가 변경된다면, 그때 즉각 응답가능
데이터가 변경되는 즉시 응답을 받을 수 있다.
regular polling과 비교했을때, 트래픽이 적다.

단점으로는, 연결을 계속 유지해야 된다는 점이다.

### 3. Server Sent Event(SSE)
SSE는 서버가 클라이언트에게 지속적으로 데이터를 전송하는 방법이다.
처음에 클라이언트가 req를 보내며, 서버와 연결함.
서버는 그 이후에, 지속적으로 데이터를 전송함.
클라이언트는 계속 날라오는 데이터(이벤트)를 처리함.

연결이 끊어지면, 클라이언트가 계속 요청을 보낸다.
푸시알림, 주식 실시간 업데이트 등에서 사용할 수 있음.

하지만, 클라이언트에서 서버로 데이터를 전송해야할 경우, 별도의 http 통신을 해야한다.
많은 클라이언트와의 지속적인 연결을 유지할 때 제한이 있을 수 있음.

## 웹소켓
http가 단방향 프로토콜 이라면, 웹소켓은 양방향 통신 프로토콜이다.
http와 함께, OSI 7계층에 위치하고 있음.
TCP의 양방향 전이중 통신을 사용하여 transport layer에 의존함. (4계층)

### 웹소켓 동작 과정
1. HandShake
2. Data trnasfer
3. Connection Termination


#### 1. HandShake
HTTP handshake를 통해 시작됨.클라이언트가 서버에 웹소켓 연결 요청, 서버는 승인
client request의 http header에는 websocket protocol로 업그레이드 하는 요청을 함.
서버는 101번 메세지로, 업그레이드를 보임.

#### 2. DataTransfer
이제 tcp처럼 전이중통신이 가능해짐.
url은 http,https대신 ws,wss를 사용함.
데이터는 '프레임'단위로 전송함. 
프레임은 header와 payload로 구성됨.

나중에 더 공부할 부분
- 프레임의 유형
- Mask
- Ping/Pong

#### 3.Connection Termination
서버 또는 클라이언트 한쪽이 종료 프레임을 전송함. 
수신 받은 쪽은 똑같이 종료프레임을 전송, TCP연결이 종료된다.



## HTTP vs WebSocket
#### HTTP
- Connectionless
- 요청/응답
- stateless
- 헤더가 큼
- 텍스트, 바이너리
- 실시간성 낮음
- best for Restful application
- 프레임 방식 없음
- 80(http), 443(https)
- 각 요청마다 핸드셰이크

#### WebSocket
- 연결형
- 양방향
- stateful
- 작은헤더
- 텍스트,바이너리
- 실시간성 높음
- real-time, gaming, chat app
- 프레임 단위 전송
- 80(ws://), 443(wss://)
- 초기 핸드셰이크 이후 지속적으로 연결


## WebSocket의 한계와 해결 방법
- 모든 브라우저가 웹소켓을 지원하지 않는다.
    - polling, long polling, streaming등의 대체 기술을 사용한다.
- 데이터가 너무 날것이다. 텍스트와 바이너리 프레임만 전송 할 수 있어, 구조가 정해져 있지 않음.
    - STOMP를 사용해서, 메세지를 구조화된 형태로 관리할 수 있다.
- 보안 문제
    - ws:// 대신 wss:// , wss가 ws의 암호화 (SSL, TLS)과정을 지원한다.
    - XSS 공격
        - 공격자가 클라이언트 측 스크립트 웹페이지에 삽입해서 실행하는 공격
        - HTML 특수문자를 이스케이프 처리 하는 방식으로 검증 필요
    - CSRF 공격
        - 공격자가 인증된 사용자를 대신해서 요청을 보내는 공격
        - 웹소켓 연결 설정시, CSRF 토큰을 사용해서 정당성을 확인해야함.
- 인증 및 권한 부여
    - 웹소켓 연결이 설정된 후, JWT 토큰과 같은 인증토큰을 사용해서 클라이언트의 신원 확인, 권한 검증 필요
- 서버자원 고갈 문제
    - 지속적인 연결을 유지학 ㅣ떄문에, 공격자가 웹소켓을 서버가 감당하기 힘들만큼 많이 열어서 서버 자원이 고갈될 수 있음.
    - 따라서 연결수를 제한해야함.
    - 일정시간동안 통신 없으면 연결을 종료해야함.
